# Analysis of Agent Observability

In the Braintrust dashboard, I explored the traces generated by the agent when answering different questions. Each user query creates a trace that contains multiple spans representing different steps in the workflow. For example, I can see the root span for the agent invocation, followed by LLM-related spans and tool call spans such as DuckDuckGo search. In some cases, the agent performs multiple tool calls within a single trace, especially when the first search does not provide enough useful information. This makes it easier to understand how the agent iteratively refines its response.

From the metrics view, I observed several useful indicators such as latency, token usage, and cost. Queries that involve more steps, like multiple tool calls or longer responses, tend to have higher latency and consume more tokens. The dashboard also shows P50 and P95 latency, which helps compare performance across different runs. Overall, these metrics provide a clear picture of how expensive and time-consuming each query is.

One thing I found interesting is that the agent can decide on its own whether to call tools again based on the quality of the intermediate results. For example, when asking about recent AI news, the agent performed multiple searches before generating the final answer. This shows that the agent is not just responding once, but actually adapting its behavior. The observability tools in Braintrust make this process visible, which is very helpful for debugging and understanding how the system works internally.
